    static environment vs dynamic environment

    Note

    static mean what type , and type-check before program runs dynamic mean what value , and evaluation as program runs

    An ML program is a sequence of bindings. Each binding gets type-checked and then (assuming it type-checks) evaluated. What type (if any) a binding has depends on a static environment, 1 which is roughly the types of the preceding bindings in the file. How a binding is evaluated depends on a dynamic environment, which is roughly the values of the preceding bindings in the file. When we just say environment, we usually mean dynamic environment. Sometimes context is used as a synonym for static environment.

    Note

    ML 程序是一系列绑定。每个绑定都会经过类型检查，然后（假设它通过类型检查）进行求值。绑定的类型（如果有）取决于静态环境，静态环境大致相当于文件中前面绑定的类型。绑定的求值方式取决于动态环境，动态环境大致相当于文件中前面绑定的值。当我们只说环境时，我们通常是指动态环境。有时上下文用作静态环境的同义词。

    ML 程序是一系列绑定，类似于其他编程语言中的一系列指令。

    每个绑定由变量名和定义其值的表达式组成。

    对每个绑定执行类型检查以确定其类型。如果绑定通过类型检查，则对其进行求值。

    绑定的类型取决于静态环境，静态环境由文件中前面绑定的类型组成。

    绑定的求值取决于动态环境，动态环境由文件中前面绑定的值组成。

    当在不指定静态或动态的情况下使用术语“环境”时，它通常是指动态环境。

    有时，“上下文”一词用作“静态环境”的同义词。

    在典型的 ML 程序中，绑定类型的决定取决于文件中前面绑定的类型，而绑定求值则取决于文件中前面绑定的值。

    形式上，我们可以将静态环境定义为从变量名到其类型的映射，将动态环境定义为从变量名到其值的映射。绑定的环境是绑定发生时的静态环境和动态环境的组合。

    类型检查绑定意味着检查绑定的类型在静态环境的上下文中是完善的。这可确保绑定在语法上正确，并且它符合语言的类型系统。

    绑定的求值意味着在动态环境的上下文中计算绑定的值。这涉及求值绑定中出现的任何表达式，并根据语言的语义组合它们的值。

    静态环境和动态环境之间的区别对于理解 ML 程序如何执行非常重要。类型检查在程序执行之前静态地执行，而求值在程序运行时动态地执行。这使得 ML 程序是类型安全的，这意味着它们不会因类型错误而产生运行时错误。

    下面是一个简单的 ML 程序示例：

    let x = 1; let y = 2; let z = x + y; 该程序的静态环境是： x: int y: int z: int 该程序的动态环境是： x: 1 y: 2 z: 3

    针对静态环境对 x 的绑定进行类型检查，发现它是完善的。针对静态环境对 y 的绑定进行类型检查，也发现它是完善的。针对静态环境对 z 的绑定进行类型检查，同样发现它是完善的。

    针对动态环境对 x 的绑定进行求值，发现它的值为 1。针对动态环境对 y 的绑定进行求值，发现它的值为 2。针对动态环境对 z 的绑定进行求值，发现它的值为 3。

    求值该程序的结果是值 3，它是 z 绑定的值。

    ML rules for expressions

    every kind of expression has three question - what is the syntax - what is the type-checking rules - what is the evalution rules

    think about

    1. variables

    2. addition (type int + type bool have no type-check)
      value -> expression but expression !-> value

    3. condition : 

          synatax :
        if e1 then e2 else e3
        where if , then , and else are keywords and 
        e1 , e2 ,e3 are subexpressions
    type-checking :
        first e1 must have type bool 
        e2 and e3 can have any type (let's call it t), ut they
        must have the same type t
        the type of the entire expression is also t
    evaluation rules:
        first evaluate e1 to a value call it v1 
        if it's true evaluate e2 and that result is the whole expression 's ressult 
        else evaluate e3 and that result is the whole expresssion's result
